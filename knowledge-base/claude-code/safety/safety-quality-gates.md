# Pre-Execution Safety Validation

## Concept

Every Claude Code command generated by an orchestrator should pass through validation gates before execution. This is the last line of defense against:
- Commands without project scope (could modify wrong directory)
- Commands without budget limits (could run up costs)
- Dangerous operations embedded in prompts
- Accidental secret exposure in prompt text

## Gate Categories

### Critical Gates (Block Execution)
1. **Scope validation**: `--cwd` must be present and point to an existing directory
2. **Budget validation**: `--max-budget-usd` must be present
3. **Command format**: Must start with `claude`

### Advisory Gates (Warn, User Override)
4. **Dangerous patterns**: rm -rf, drop table/database, force push, --force, git reset --hard
5. **Secret detection**: password, api_key, secret_key, access_token in prompt text
6. **Model appropriateness**: opus for trivial tasks (wasteful), haiku for complex ones (insufficient)

## Integration Pattern

```
User input → Task routing → LLM generates command → Quality gates → User confirmation → Execution
```

Gates run after the LLM generates the command but before the user sees the confirmation prompt. Critical issues prevent the confirmation prompt from appearing. Warnings are displayed alongside the command.

## Relationship to Safety Chunks

Quality gates complement (not replace) the safety chunks that are always injected into the system prompt:
- **Safety chunks**: Guide the LLM to generate safe commands in the first place
- **Quality gates**: Validate the generated command as a second check
- Together: defense in depth
